# CoCoFlow Chest Samples

Chest Samples demonstrates the two persistence paths working together:

- Context path: a chest is an `ItemContext` entity. `PersistenceContext` snapshots and restores whether it is available, opening, opened, or consumed.
- Container path: rewards, inventory contents, event log, and world facts live in `PersistenceContainerStore` records and are saved through `PersistenceContainerSection`.

## Contents

- `Scripts/Runtime/Components/ChestSampleSceneInstaller.cs`: creates a small runtime `PersistenceContainerCatalog`, assigns it to the scene `PersistenceContainerStore`, and materializes starter containers.
- `Scripts/Runtime/Components/ChestSampleOpenDriver.cs`: tiny interaction entry point that calls `ItemInputDriver.RequestOpen`.
- `Scripts/Runtime/States`: sample chest states for `Available -> Opening -> Opened`.

## Scene Wiring

Create one scene root object:

- `PersistenceContainerStore`
- `ChestSampleSceneInstaller`

Create one chest object:

- `ItemContextProvider`
- `ItemInputDriver`
- `ItemLifeCycle`
- `PersistenceContext`
- `PersistenceContainerBridge`
- `CoCoStateController`
- `CCS_Chest_Available`
- `CCS_Chest_Opening`
- `CCS_Chest_Opened`
- `ChestSampleOpenDriver`

Wire the state controller with one `Main` State Layer:

- Default state: `CCS_Chest_Available`
- Available states: `CCS_Chest_Available`, `CCS_Chest_Opening`, `CCS_Chest_Opened`
- Context Provider: the same `ItemContextProvider`

## Flow

1. `ChestSampleOpenDriver.OpenChest` writes open intent into `ItemContext`.
2. `CCS_Chest_Available` sees the intent and transitions to `CCS_Chest_Opening`.
3. `CCS_Chest_Opening` sets the context to `Opening`, dispatches a reward command through `PersistenceContainerBridge`, then sets a world fact and event log entry.
4. `CCS_Chest_Opened` keeps the item context in `Opened`.

After saving and loading:

- The chest reopens through the Context path because `PersistenceContext` restores `ItemContext.ItemState`.
- The granted reward and world fact remain through the Container path because `PersistenceContainerStore` restores the container section.

## Notes

- This sample intentionally does not include a prefab yet; it keeps the wiring explicit so the two persistence paths are easy to inspect.
- The runtime catalog is generated by `ChestSampleSceneInstaller` for a minimal sample. Production projects should use a real `PersistenceContainerCatalog` asset.
- Network forwarding is separate. Use Network Samples' `PersistenceNetworkContainerEventBridge` if container commands need to cross a transport.
